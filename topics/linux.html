Basic Git Commands and Management

Added: 2025-12-03 12:32

This document provides an overview of essential Git commands for managing repositories, branches, remotes, and changes. It covers basic operations such as initializing repositories, staging and committing changes, and reverting or resetting commits.

Key Points

  • Git commands are essential for version control and collaboration.
  • Branching and merging are crucial for managing different lines of development.
  • Stashing allows temporary saving of changes without committing.

Commands

CommandDescription
git init Initializes a new Git repository in the current directory.
git clone <repo_url> Clones a repository from a remote URL to your local machine.
git status Shows the working directory status (which files are modified, staged, etc.).
git add <file> Stages a file or files to be committed.
git commit -m "message" Commits staged changes to the local repository with a message.
git push Pushes committed changes to the remote repository.
git pull Fetches and merges changes from the remote repository to your local working directory.
git fetch Downloads objects and refs from another repository (without merging).
git merge <branch_name> Merges changes from the specified branch into the current branch.
git log Shows the commit history of the repository.
git branch Lists all branches in your repository.
git checkout <branch_name> Switches to an existing branch.
git remote -v Shows the remote repository URLs.
git stash Stashes the changes in your working directory (both staged and unstaged).
git reset <commit_id> Moves HEAD to a specific commit, leaving your working directory intact.
git revert <commit_id> Creates a new commit that undoes the changes of the specified commit.
Always commit changes with meaningful messages for better tracking.
Use branches for new features or bug fixes to keep the main branch stable.

‚Ä¢ Regularly push changes to the remote repository to avoid data loss.

‚Ä¢ Review changes with 'git diff' before committing to ensure accuracy.

Git Stash, Tags, Config, Aliases, Hooks, Submodules, Workflow Commands, Merge Strategies, Bisect, Clean and Prune

Added: 2025-12-03 12:32

This section covers various Git commands and concepts including stashing changes, managing tags, configuring Git settings, creating aliases, using hooks for automation, handling submodules, and managing collaboration through workflow commands. Additionally, it discusses merge strategies, the bisect process for bug tracking, and cleaning up untracked files.

Key Points

  • Stashing allows you to save changes temporarily without committing.
  • Tags are used to mark specific points in the repository's history.
  • Git hooks automate tasks during Git actions.

Code Examples

Stash changes with a message

git stash save "message"

Create a new tag at the current commit

git tag <tag_name>

Set global username for commits

git config --global user.name "Your Name"

Create a custom Git alias for checkout

git config --global alias.co checkout

Start a binary search for the commit that introduced a bug

git bisect start

Run garbage collection to clean up unnecessary files

git gc

Commands

CommandDescription
git stash pop Applies the latest stash and removes it from the stash list.
git stash apply Applies a stash without removing it from the list.
git stash drop Removes a specific stash from the stash list.
git stash clear Clears all stashes.
git tag -a <tag_name> -m "message" Creates an annotated tag with a message.
git push origin <tag_name> Pushes a specific tag to the remote repository.
git config --list Lists all configuration settings.
git config --global core.editor Sets the default text editor for Git.
git pull --rebase Fetches and applies remote changes, rebasing your local commits on top of them.
git merge --no-ff Merges a branch with a 'no fast-forward' option.
git bisect bad Marks the current commit as 'bad'.
git clean -f Removes untracked files from the working directory.
git prune Removes objects that are no longer needed.
Use stashing to save work in progress before switching branches.
Regularly clean up your repository to maintain performance.

‚Ä¢ Always provide meaningful messages when creating tags.

‚Ä¢ Use hooks to enforce coding standards and automate testing.

Git Commands for DevOps and CI/CD

Added: 2025-12-03 12:32

This section covers essential Git commands and practices for DevOps, focusing on Continuous Integration/Continuous Deployment (CI/CD), Infrastructure as Code (IaC), and deployment automation. It highlights how Git can be utilized to manage code, automate processes, and ensure stable deployments across multiple environments.

Key Points

  • Git commands facilitate CI/CD processes by managing code versions and automating deployments.
  • Infrastructure as Code (IaC) can be effectively managed using Git for version control.
  • Git hooks and feature flags enhance automation and control over deployments.

Commands

CommandDescription
git archive --format=tar --output=<output_file>.tar <branch_name> Creates a tarball archive of the repository at a specific branch.
git archive --format=zip --output=<output_file>.zip <branch_name> Creates a zip archive of the repository at a specific branch.
git blame <file_path> Shows who last modified each line of a file and when.
git annotate <file_path> Displays commit information for each line of a file.
git reflog Displays a log of all movements of the HEAD pointer.
git fetch --all Fetches all branches from all remotes.
git merge origin/ Merges the latest changes from a remote branch into the local branch.
git push --force-with-lease Pushes changes forcefully while ensuring the remote branch hasn‚Äôt changed unexpectedly.
git rebase origin/ Rebases your branch to ensure it is up to date with the base branch.
git tag <version_number> Tags a specific version of the code.
git clone <IaC_repo_url> Clones repositories that contain configuration files or infrastructure code.
git checkout Switches to a branch that contains a specific environment‚Äôs infrastructure code.
git pull origin <branch_name> Pulls the latest changes to infrastructure code.
git submodule Manages shared code or dependencies across multiple infrastructure repositories.
git pre-commit Automatically runs tests or linters on code before it‚Äôs committed.
git post-merge Triggers deployment processes or runs tests after a merge.
git pre-push Runs a script to test whether the pushed code will break something in production.
git commit-msg Enforces commit message conventions.
git branch <feature_flag_name> Creates a branch dedicated to developing a specific feature.
git checkout -b <feature_flag_branch> Creates a new branch to work on a feature behind a feature flag.
git merge <feature_flag_branch> Merges a completed feature flag branch into the main codebase.
git checkout <environment_branch> Switches to the branch corresponding to the environment you are deploying to.
git merge --no-ff <feature_branch> Merges feature branches into specific environment branches with a merge commit.
git tag <release_version> Tags releases when promoting code to staging or production environments.
git push origin Triggers CI/CD pipelines that automatically deploy code.
git reset --hard Resets the working directory to a known state before deploying.
Always fetch the latest changes before merging to avoid conflicts.
Use descriptive commit messages to maintain clarity in version history.

‚Ä¢ Use feature flags to manage the deployment of new features safely.

Advanced Git for Managing CI/CD Pipelines

Added: 2025-12-03 12:32

This section covers the integration of Git with CI/CD systems, emphasizing how Git commands can trigger build and deployment pipelines. It also discusses the importance of managing commit history and resolving merge conflicts in a collaborative development environment.

Key Points

  • Git commands can trigger CI/CD pipelines for deployments.
  • Managing commit history is crucial for maintaining a clean project history.
  • Merge conflicts can be resolved using Git's built-in tools.

Commands

CommandDescription
git push --tags Triggers a CI/CD pipeline that looks for tags to start a deployment.
git fetch --tags Fetches the latest tags to ensure the correct version of the code is deployed.
git log --oneline --graph Visualizes commit histories during pipeline execution.
git diff <commit_id> <commit_id> Shows changes between commits to decide on triggering builds.
git clone <repo_url> Clones a repository containing a Dockerfile and application code.
git checkout Switches to a branch where different Docker images or configurations might be defined.
git pull origin <branch_name> Pulls the latest code from the branch and rebuilds Docker images.
git tag Tags a version of the Docker image.
git rm --cached Removes a file from version control.
git commit --amend Amends the last commit to remove sensitive information.
git rebase -i <commit_id> Starts an interactive rebase to clean up commit history.
git rebase --onto Rebases a branch onto a new base commit.
git rebase --skip Skips the current commit while rebasing.
git rebase --continue Continues the rebase process after resolving conflicts.
git mergetool Opens an external merge tool to help resolve conflicts.
git merge --abort Aborts the merge process and restores the previous state.
git diff <branch_1> <branch_2> Shows differences between two branches to identify conflicts.
git status Displays files in conflict that need manual resolution.
git config --global user.signingkey <key_id> Configures the GPG key for signing commits.
git commit --gpg-sign Signs a commit with your GPG key.
git tag -s <tag_name> Creates a signed tag for important release points.
Regularly fetch tags to keep your CI/CD pipeline updated.
Use interactive rebase to maintain a clean commit history before merging.

‚Ä¢ Keep sensitive information out of version control using .gitignore.

‚Ä¢ Resolve merge conflicts promptly to maintain workflow efficiency.
